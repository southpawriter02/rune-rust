## v0.4.2a: The Repute (Faction System)

**Goal:** Implement the `FactionService` to track the player's standing with various groups in the world. This establishes the "Friend or Foe" dynamic beyond simple combat aggression.

### 1. Architecture & Data Flow

*   **Reputation Value:** A signed integer ranging from -100 (Hated) to +100 (Exalted).
    *   **Hated (-100 to -50):** Attack on sight, no trade.
    *   **Hostile (-49 to -10):** Refuse to talk, threatening barks.
    *   **Neutral (-9 to 9):** Indifferent, basic trade.
    *   **Friendly (10 to 49):** Better prices, extra dialogue options.
    *   **Exalted (50 to 100):** Unique items, special quests.
*   **Faction Definitions:**
    *   **Iron-Banes:** Scavengers of the upper levels.
    *   **Dvergr:** Deep-dwelling master smiths.
    *   **The Bound:** Cultists of the Glitch (Always Hostile to most).
    *   **The Faceless:** Mysterious traders.

**Data Flow: Reputation Change**
1.  **Trigger:** Event occurs (Quest Complete, Enemy Killed, Dialogue Choice).
2.  **Modification:** `FactionService.ModifyReputation(FactionType.IronBanes, +5)`.
3.  **Clamp:** Value clamped between -100 and +100.
4.  **Disposition Check:** If the new value crosses a threshold (e.g., Neutral -> Friendly), trigger a `DispositionChangedEvent`.
5.  **Persistence:** Save `Dictionary<FactionType, int>` in `Character` entity.

### 2. Logic Decision Trees

**A. Get Disposition Input: Reputation Value**
1.  Check Range:
    *   `<= -50`: Return `Disposition.Hated`.
    *   `<= -10`: Return `Disposition.Hostile`.
    *   `<= 9`: Return `Disposition.Neutral`.
    *   `<= 49`: Return `Disposition.Friendly`.
    *   `>= 50`: Return `Disposition.Exalted`.

**B. Modify Reputation Input: Faction, Amount**
1.  Get current value.
2.  Add amount.
3.  Clamp to Min/Max.
4.  If Tier changed, fire event.

{
    public void ModifyReputation(Character character, FactionType faction, int amount)
    {
        var current = character.GetReputation(faction);
        var next = Math.Clamp(current + amount, -100, 100);
        character.SetReputation(faction, next);

        // Log and Event logic...
    }

    public Disposition GetDisposition(int reputation)
    {
        return reputation switch
        {
            <= -50 => Disposition.Hated,
            <= -10 => Disposition.Hostile,
            <= 9 => Disposition.Neutral,
            <= 49 => Disposition.Friendly,
            _ => Disposition.Exalted
        };
    }
}

## v0.4.2b: The Voice (Dialogue Engine)

**Goal:** Implement a branching dialogue system (`DialogueService`) where conversation options can be locked or unlocked based on Skills, Attributes, or Faction Reputation.

### 1. Architecture & Data Flow

*   **Dialogue Tree:** A graph of `DialogueNode` objects.
*   **Node Structure:**
    *   `Id`: Unique identifier.
    *   `Text`: The NPC's spoken line (Narrative flavor).
    *   `Options`: List of `DialogueOption` (Player responses).
*   **Option Structure:**
    *   `Text`: The player's choice text.
    *   `NextNodeId`: ID of the node to jump to.
    *   `Conditions`: Requirements to see/pick this option (e.g., `WITS >= 6`, `IronBanes >= Friendly`).
    *   `Effects`: Actions to take on selection (e.g., `ModifyReputation`, `GiveItem`, `StartQuest`).

**Data Flow: Conversation Step**
1.  **Start:** `DialogueService.StartDialogue(npcId)`.
2.  **Load:** Fetch `DialogueTree` from repository.
3.  **Filter Options:** For the current node, iterate through `Options`.
    *   Evaluate `Condition`. If false, either hide the option or mark it as [LOCKED] (grayed out).
4.  **Present:** Return `DialogueView` model to UI.
5.  **Select:** Player chooses Option X.
6.  **Execute:** Run `Effects` (if any).
7.  **Transition:** Set Current Node to `NextNodeId`.

### 2. Logic Decision Trees

**A. Option Availability Check**
1.  Iterate `Conditions`.
2.  **Attribute Check:** Is Player Attribute >= Required?
3.  **Skill Check:** Is Player Skill >= Required?
4.  **Reputation Check:** Is Faction Rep >= Required?
5.  **Flag Check:** Has Player met global flag X?
6.  **Result:**
    *   Pass: Option is Selectable.
    *   Fail (Visible): Option shown but disabled with tooltip "Requires WITS 6".
    *   Fail (Hidden): Option not rendered.

## v0.4.2c: The Parley (UI & Integration)

**Goal:** Integrate the dialogue system into the Game Loop and provide a rich TUI using Spectre.Console.

### 1. Architecture & Data Flow

*   **Game State:** Introduce `GamePhase.Dialogue`.
*   **Command Interception:** When in `Dialogue` phase, standard commands (North, Attack) are suspended. Input is routed to `DialogueController`.
*   **UI Layout:**
    *   **Top Panel:** NPC Name and Portrait (ASCII).
    *   **Center Panel:** NPC Text (Typewriter effect).
    *   **Bottom Panel:** Selection Prompt for Player Options.

### 2. Logic Decision Trees

**A. Talk Command Input: Target NPC**
1.  Validate target is in room.
2.  Validate target is not Hostile (Combat triggers instead).
3.  **Transition:** Switch `GameState` to `Dialogue`.
4.  **Initialize:** Load Root Node of NPC's tree.
5.  **Render:** Call `DialogueRenderer`.
