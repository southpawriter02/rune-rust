# v0.3.23b: The Loop (Event-Driven Architecture) - Expanded Implementation Plan

## Overview

Transform the main game loop from a blocking "Wait for Key" model to a non-blocking, Event-Driven Architecture. This allows the game to process background tasks (Ambience, VFX, Animations) while waiting for user input, enabling a more dynamic "alive" feel without sacrificing the turn-based core.

**Parent Plan:** [v0.3.23](./v0.3.23.md)
**Status:** Planned

---

## Architecture Summary

### Key Design Decisions

1. **Non-Blocking Loop:** The main `while` loop checks `_inputService.IsInputAvailable()` instead of blocking on `ReadKey`.
2. **Tick Rate:** The loop runs at ~50 ticks per second (`Task.Delay(20)`), providing sufficient resolution for UI animations and input responsiveness.
3. **Dirty Flag Rendering:** To avoid expensive console redraws every tick, we introduce a `RenderRequired` flag. Rendering only occurs when state changes (Input processed) or Visual Effects update.
4. **Cancellation Token:** Define a `CancellationTokenSource` for the entire game session to handle graceful shutdowns (Quit command, SIGINT).

### The Game Loop Lifecycle

```
START
  |
  v
WHILE ( !Quit && !CancellationRequested )
  |
  +-- [1. Input Phase]
  |   Is Input Available?
  |   Yes: 
  |     Event = ReadNext()
  |     ProcessEvent(Event) -> Updates Game Action logic
  |     RenderRequired = True
  |
  +-- [2. Update Phase]
  |   Update Ambience (Tick)
  |   Update VFX State
  |   If (VFX Active) -> RenderRequired = True
  |
  +-- [3. Render Phase]
  |   If (RenderRequired)
  |     Draw Current View (Combat/Exploration)
  |     RenderRequired = False
  |
  +-- [4. Throttle]
  |   await Task.Delay(20)
  |
END
```

```csharp
public class GameService : IGameService
{
    // ... Dependencies ...
    private bool _renderRequired = true;

    public async Task StartAsync(CancellationToken ct = default)
    {
        // Initialization...
        _inputHandler.DisplayMessage("Welcome...");

        while (_state.Phase != GamePhase.Quit && !ct.IsCancellationRequested)
        {
            bool inputProcessed = false;

            // 1. Process Input (Non-blocking)
            if (_inputService.IsInputAvailable())
            {
                var inputEvent = _inputService.ReadNext();
                await ProcessInputEventAsync(inputEvent); // Logic moved from ParseAndExecute
                inputProcessed = true;
                _renderRequired = true; // Input usually changes state
            }

            // 2. Background Updates
            // Check if VFX are running (future: IVisualEffectService.IsAnimating)
            // If animation running, set _renderRequired = true

            // Trigger Ambience Tick if needed (currently purely turn-based, so skipped for now)

            // 3. Render
            if (_renderRequired)
            {
                RenderCurrentPhase();
                _renderRequired = false;
            }

            // 4. Throttle
            if (!inputProcessed)
            {
                await Task.Delay(20, ct); 
            }
        }
        
        // ... Shutdown logic
    }

    private void RenderCurrentPhase()
    {
         if (_state.Phase == GamePhase.Combat) 
             _combatRenderer?.Render(_combatService.GetViewModel());
         else if (_state.Phase == GamePhase.Exploration)
             // ... Build and Render Exploration VM
    }
}
```

### 3. VisualEffectService Redraw Trigger

To ensure VFX flashes (which happen asynchronously) trigger a redraw, `VisualEffectService` needs to signal the loop.

**Option A:** Polling. `GameService` checks `_vfxService.IsActive` every tick.
**Option B (Preferred):** Event. `VisualEffectService` fires `EffectStarted` / `EffectEnded`.

**Refactor `VisualEffectService.cs`:**
Add `public event Action? OnInvalidateVisuals;`
Invoke this when `SetBorderOverride` and `ClearBorderOverride` are called.

**Update `GameService.cs`:**
Subscribe to `visualEffectService.OnInvalidateVisuals += () => _renderRequired = true;`

## Verification Strategy

### 4.1. Responsiveness Test
- **Goal:** Ensure the non-blocking loop processes keys immediately.
- **Action:** Hold down a movement key.
- **Expected:** Character moves smoothly without input building up or lag.

### 4.2. CPU Usage Test
- **Goal:** Ensure `Task.Delay(20)` effectively throttles the loop.
- **Action:** Open Activity Monitor / Task Manager while game is idle at prompt.
- **Expected:** CPU usage should be minimal (< 5%), distinct from a `while(true)` tight loop.

### 4.3. VFX Timing Test
- **Goal:** Ensure border flashes are rendered immediately.
- **Action:** Trigger a combat hit (or debug effect).
- **Expected:** The border changes color instantly (due to `OnInvalidateVisuals` triggering a Render) and clears after delay.

## Critical Adjustments to v0.3.23a
- `InputService` MUST serve as the sole source of input.
- `GameService` must NOT call `_inputHandler.GetInput()` (blocking) anymore.
- `CommandParser` might need adjustment to accept `InputEvent` or mapping logic needs to be lifted into `GameService`. *Correction:* `CommandParser` parses strings. We map `InputEvent` -> `GameAction` -> String command (internal) OR refactor `CommandParser` to handle `GameAction` directly (Better).
- **Decision:** For this version, map `GameAction` back to string command (e.g. `GameAction.MoveNorth` -> "north") to utilize existing `CommandParser` logic without huge refactor. We can modernize `CommandParser` in v0.3.25.
