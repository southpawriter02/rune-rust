## Summary

Version 0.3.25d completes the v0.3.25 "Externalized Capture Templates" initiative by creating comprehensive documentation that enables content authors to add new capture templates without programming knowledge. This release introduces a 444-line authoring guide covering template creation workflows, Domain 4 compliance requirements, AAM-VOICE Layer 2 guidelines, and troubleshooting procedures, alongside updates to the capture templates README with quick reference tables and corrected documentation links. Changes span the **Documentation Layer** exclusively (guides, reference documentation). Key content includes a 5-step Quick Start workflow, annotated template examples with field-by-field breakdowns, Domain 4 forbidden pattern tables with medieval measurement equivalents, AAM-VOICE translation guides, NotebookLM-to-JSON conversion workflows, and a comprehensive FAQ section. This documentation empowers lore writers, content designers, and world-builders to contribute to the knowledge system by editing JSON files directly.

### NotebookLM to JSON Workflow

**Lines:** 290-338

**Complete 5-Step Process:**

#### 1. Export from NotebookLM (Lines 294-298)
- Open NotebookLM source
- Copy lore text
- Note context (creature, location, discovery method)

#### 2. Create Template Skeleton (Lines 300-312)
```json
{
  "id": "",
  "type": "",
  "fragmentContent": "",
  "source": "",
  "matchKeywords": [],
  "quality": 15,
  "tags": []
}
```

#### 3. Fill In Fields (Lines 314-322)
1. **id:** Create from context: `"blight-fungal-network"`
2. **type:** Match discovery: examining creature = `"Specimen"`
3. **fragmentContent:** Paste and refine for Domain 4 compliance
4. **source:** Where found: `"Biological analysis"`
5. **matchKeywords:** Extract key nouns: `["blight", "fungal", "creature"]`
6. **tags:** Add metadata: `["horror", "mutation"]`

#### 4. Domain 4 Pass (Lines 324-331)
Checklist:
- [ ] No percentages or decimals
- [ ] No metric measurements
- [ ] No modern terminology
- [ ] Sensory descriptions only
- [ ] Medieval vocabulary

#### 5. Validate and Commit (Lines 333-338)
```bash
pwsh scripts/validate-templates.ps1
git add data/capture-templates/
git commit -m "feat(lore): add blight-fungal-network template"
```

**Workflow Purpose:**
- Bridges gap between prose drafting (NotebookLM) and structured data (JSON)
- Provides clear conversion methodology
- Integrates Domain 4 compliance checking
- Supports existing writer toolchains

### Troubleshooting Section

**Lines:** 341-389

**4 Common Errors with Solutions:**

#### 1. Missing 'id' Error (Lines 344-352)
**Error Message:**
```
Error: ancient-ruin.json: Template missing 'id'
```

**Fix:**
```json
"id": "ruin-your-template-name"
```

#### 2. Invalid Type Error (Lines 354-367)
**Error Message:**
```
Warning: Template 'my-template' has invalid type 'TextFrag'
```

**Fix:** Use exact enum values (case-sensitive):
- `TextFragment`
- `EchoRecording`
- `VisualRecord`
- `Specimen`
- `OralHistory`
- `RunicTrace`

#### 3. Domain 4 Violation Warning (Lines 369-375)
**Error Message:**
```
Warning: Template 'ruin-measurements' may violate Domain 4 (precision measurements)
```

**Fix:** Review fragmentContent for percentages, decimals, metric units. Replace with qualitative descriptions.

#### 4. JSON Parse Error (Lines 377-389)
**Error Message:**
```
Error: ancient-ruin.json: JSON parse error - Unexpected token
```

**Possible Causes:**
- Missing comma between templates
- Missing closing brace `}`
- Unescaped quotes in text (use `\"` or `'`)
- Trailing comma after last template

**Tip:** Use JSON validator like jsonlint.com

## Technical Implementation Notes

### 1. Documentation Architecture

**Design Philosophy:** Progressive Disclosure

**Information Hierarchy:**
```
Level 1: README.md (Quick Reference)
  ↓
Level 2: capture-template-authoring.md (Comprehensive Guide)
  ↓
Level 3: sample-codex-entries.md (Content Examples)
  ↓
Level 4: SPEC-CODEX-001.md (Technical Specification)
```

**Navigation Flow:**
1. Content author opens `data/capture-templates/` directory
2. Reads README.md blockquote directing to authoring guide
3. Authoring guide references prerequisite docs (samples, spec)
4. Guide provides step-by-step workflow with inline examples
5. Author returns to README for quick reference during work

**Benefits:**
- Minimizes cognitive load (quick start in 5 steps)
- Supports learning curve (basic → advanced)
- Reduces context switching (inline examples vs. external links)
- Enables self-service (FAQ, troubleshooting sections)

### 5. Voice & Tone Standards

**Target Audience:** Non-programmers (lore writers, content designers, world-builders)

**Writing Principles:**

1. **Imperative Instructions:**
   - "Open `data/capture-templates/categories/`" (not "You should open...")
   - "Run validation: `pwsh scripts/validate-templates.ps1`" (not "You can run...")

2. **Active Voice:**
   - "Add a new template object" (not "A new template object should be added")
   - "The system links them to entries" (not "They are linked to entries by the system")

3. **Concrete Examples:**
   - Every rule has at least one example
   - Every error has a fix with code snippet
   - Every table has real data, not placeholders

4. **Plain Language:**
   - "Forbidden" instead of "prohibited"
   - "Fix" instead of "remediate"
   - "Template" instead of "data structure"

5. **Encouraging Tone:**
   - "No programming required" (Line 4)
   - FAQ answers avoid "no" when possible
   - Troubleshooting provides solutions, not just error descriptions
