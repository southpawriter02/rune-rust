# v0.3.23c: The Mouse (TUI Mouse Support)

**Status:** Planned
**Parent Plan:** [v0.3.23](./v0.3.23.md)
**Context:** This sub-plan adds experimental mouse support to the TUI. It requires a terminal that supports VT mouse reporting (e.g., Windows Terminal, iTerm2, Alacritty).

## 1. Terminal Configuration

### 1.1. Enable Mouse Reporting
In `RuneAndRust.Terminal/Services/TerminalService.cs` (or initialization logic):
- Send the VT sequence to enable mouse tracking.
- Common sequence: `\x1b[?1003h` (Any event mouse tracking) or `\x1b[?1000h` (Click tracking).
- Use Spectre.Console if it exposes this, or raw `AnsiConsole.Write`.

```csharp
// Example (check Spectre docs for native support first)
AnsiConsole.Write(new ControlCode("\x1b[?1003h"));
```

### 1.2. Cleanup
Ensure the sequence `\x1b[?1003l` (disable) is sent on application exit to restore terminal state.

## 2. Input Parsing

### 2.1. Update InputService
The `InputService` needs to handle the VT sequences generated by mouse events.
- These sequences usually look like `\x1b[M<btn><x><y>`.
- `Console.ReadKey` might split these into multiple keys.
- **Challenge:** Reading escape sequences reliably without blocking `Esc` key logic.

**Strategy:**
- If `Console.KeyAvailable` is true, read key.
- If key is `Escape`, check if more keys are immediately available.
- If so, read them to form a sequence.
- Parse the sequence into a `MouseEvent`.

### 2.2. Implement Parsing Logic
Add a private method `ParseMouseSequence()` in `InputService`.
- Map X/Y coordinates (offset might be 32 + 1).

## 3. Interaction Mapping

### 3.1. Combat Grid Interaction
Update `RuneAndRust.Terminal/Rendering/CombatGridRenderer.cs` (or the input handler for combat).
- When a `MouseEvent` is received:
    - Check if (X, Y) falls within the Combat Grid bounds.
    - Convert Screen (X, Y) to Grid (Row, Col).
    - If valid, treat as "Selection" or "Inspect".
    - If Clicked, trigger `GameAction.Confirm` on that cell.
