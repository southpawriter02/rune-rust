v0.3.6c: The Telegraph (Intent & Status)
Goal: Update CombatService to pre-calculate enemy actions at the start of a round and render Intent/Status icons on the Tactical Grid based on player perception stats.

1. Architecture & Data Flow
The Planning Phase Currently, AI decides its move on its turn. To implement Telegraphing, we must split AI logic into Planning and Execution.
	1	Round Start: CombatService calls EnemyAIService.PlanActions() for all living enemies.
	2	Storage: The resulting CombatAction is stored in Combatant.PlannedAction.
	3	Analysis Check: The engine compares Player WITS + Bonuses vs Enemy Stealth/Level.
	4	ViewModel Mapping:
	◦	If Check Passed (or Jötun-Reader): Map PlannedAction to Icon (e.g., ⚔️).
	◦	If Check Failed: Map to ?.
	5	Execution: When the enemy's turn arrives, it executes the stored plan rather than calculating a new one.

A. Core Layer (Combatant Update)

public class Combatant
{
    // ... existing properties ...

    // The action the AI intends to take this round
    public CombatAction? PlannedAction { get; set; }

    // Whether the player has successfully perceived the intent
    public bool IsIntentRevealed { get; set; }

    // Helper for ViewModel mapping
    public string GetStatusIcons()
    {
        var sb = new StringBuilder();
        foreach (var effect in StatusEffects)
        {
            string icon = effect.Type switch
            {
                StatusEffectType.Bleeding => "🩸",
                StatusEffectType.Poisoned => "🤢",
                StatusEffectType.Stunned => "💫",
                StatusEffectType.Vulnerable => "💔",
                StatusEffectType.Fortified => "🛡️",
                _ => "•"
            };
            sb.Append(effect.Stacks > 1 ? $"{icon}x{effect.Stacks} " : $"{icon} ");
        }
        return sb.ToString().Trim();
    }
}
B. Engine Layer (Planning Service)
File: RuneAndRust.Engine/Services/CombatService.cs
// Inside StartNewRound()
private void PlanEnemyActions()
{
    foreach (var enemy in _state.TurnOrder.Where(c => !c.IsPlayer && c.CurrentHp > 0))
    {
        // Ask AI service to decide NOW, but store it for later
        enemy.PlannedAction = _aiService.DetermineAction(enemy, _state);

        // Calculate visibility
        var player = _state.CurrentCharacter; // Wrapper or Entity
        bool isAnalyst = player.Archetype == ArchetypeType.Adept; // Simplified check

        // Jötun-Readers or Analyzed targets always reveal
        if (isAnalyst || _statusService.HasEffect(enemy, StatusEffectType.Analyzed))
        {
            enemy.IsIntentRevealed = true;
        }
        else
        {
            // Basic WITS check
            var check = _dice.Roll(player.Attributes[Attribute.Wits], "Intent Check");
            enemy.IsIntentRevealed = check.Successes >= 1; // Simple threshold
        }
    }
}

# Changelog: v0.3.6c - The Telegraph
**Release Date:** 2025-XX-XX

## Summary
The "Telegraph" update adds a critical layer of tactical depth to combat. Enemies now plan their actions at the start of the round, and observant players can predict these moves to counter them. Additionally, status effects are now visualized with intuitive icons directly on the battlefield grid.

## Features
- **Enemy Intent System:**
    - AI determines actions at the start of the round.
    - **Telegraphing:** Players perform a passive WITS check to reveal enemy intentions.
    - **Jötun-Reader Bonus:** Adept archetypes automatically perceive all enemy intents.
    - **Visuals:** Icons indicate upcoming attacks (⚔️), defense (🛡️), or buffs (✨).
- **Status Icons:**
    - Active effects are now displayed on combatant cards.
    - Examples: 🩸 (Bleeding), 🤢 (Poisoned), 💫 (Stunned).
    - Stack counts are displayed for intensifying effects (e.g., 🩸x3).

## Technical
- Refactored `EnemyAIService` to separate decision logic from execution.
- Updated `CombatGridRenderer` to support Unicode emoji rendering.
