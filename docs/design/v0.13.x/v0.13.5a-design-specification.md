## 1. Overview

### 1.1 Purpose

This document provides a comprehensive design specification for v0.13.5a, the Naming Convention Alignment implementation. This part focuses on auditing all TUI and GUI component names, establishing standardized naming conventions with consistent suffixes, renaming components that do not follow the standards, and updating all references to ensure codebase-wide consistency.

### 1.2 Scope

**In Scope:**
- Audit all TUI component names for consistency
- Audit all GUI component names for consistency
- Define and document naming convention standards
- Rename components with inconsistent names
- Update all references to renamed components
- Create naming convention documentation

**Out of Scope:**
- Theme consolidation (v0.13.5b)
- Service integration changes (v0.13.5c)
- Functional changes to components
- Error handling changes (v0.13.5d)
- Shared layer refactoring (v0.13.5e)
- Accessibility and performance (v0.13.5f)

### 1.3 Key Deliverables

| Type | Count | Details |
|------|-------|---------|
| Documentation | 1 | `NamingConventionGuidelines.md` in `/docs/guidelines/` |
| Renamed Components | ~10 | TUI/GUI components with inconsistent names |
| Reference Updates | ~20 | Files referencing renamed components |
| Unit Tests | ~4 | Naming convention validation tests |

## 4. Naming Convention Standards

### 4.1 Primary Suffix Definitions

| Suffix | Usage | Interaction | Examples |
|--------|-------|-------------|----------|
| `-Panel` | Container/layout components | Contains sub-components | `InventoryPanel`, `CombatPanel`, `CraftingPanel` |
| `-View` | Full-screen views | Top-level coordination | `CombatView`, `CharacterView`, `LeaderboardView` |
| `-Control` | Interactive elements | Accepts user input | `HealthBarControl`, `SlotControl`, `ButtonControl` |
| `-Renderer` | Output generation | Transforms data to display | `GridRenderer`, `TextRenderer`, `DiceRollRenderer` |
| `-Display` | Read-only output | Passive visualization | `HealthBarDisplay`, `ResourceBarDisplay`, `StatusDisplay` |
| `-Indicator` | Status visualization | Shows current state | `ComboIndicator`, `StanceIndicator`, `StreakIndicator` |

### 4.2 Secondary Suffix Definitions

| Suffix | Usage | Examples |
|--------|-------|----------|
| `-Dashboard` | Multi-section overview displays | `StatisticsDashboard`, `PerformanceDashboard` |
| `-Chart` | Data visualization (graphs) | `RollDistributionChart`, `DamageChart` |
| `-List` | Scrollable item lists | `InventoryList`, `AchievementList` |
| `-Grid` | Two-dimensional layouts | `CombatGrid`, `InventoryGrid` |
| `-Formatter` | String formatting utilities | `DamageFormatter`, `TimeFormatter` |
| `-Presenter` | Data preparation for display | `StatsPresenter`, `ItemPresenter` |

### 4.4 TUI vs GUI Suffix Conventions

| Context | Suffix Usage | Notes |
|---------|--------------|-------|
| TUI (Console) | `-Display` for passive bars | Console output, no direct interaction |
| GUI (Avalonia) | `-Control` for interactive bars | Mouse/keyboard interaction supported |
| Both | `-Panel`, `-View`, `-Renderer` | Shared concepts across layers |

### 4.5 Naming Convention Anti-Patterns

| Anti-Pattern | Problem | Correct Pattern |
|--------------|---------|-----------------|
| `HealthBar` (no suffix) | Ambiguous - display or control? | `HealthBarDisplay` or `HealthBarControl` |
| `CombatGridView` for panel | Misleading - not a full-screen view | `CombatGridPanel` |
| `ColoredText` (no suffix) | Unclear purpose | `ColoredTextRenderer` |
| `XyzManager` for UI | Business logic term, not UI | Use appropriate UI suffix |
| `XyzHelper` for rendering | Vague, non-descriptive | `XyzRenderer` or `XyzFormatter` |

## 18. Implementation Notes

### 18.1 Renaming Strategy

1. **Create new directory structure first** - Create `Displays/` directory before moving files
2. **Rename in dependency order** - Start with `ColoredTextRenderer` (most referenced)
3. **Update references immediately** - After each rename, update all references before proceeding
4. **Run tests after each rename** - Verify no regressions

### 18.2 IDE Refactoring

Use IDE refactoring tools where available:
- Visual Studio: "Rename" (F2) with "Preview changes"
- Rider: "Refactor > Rename" with "Search in comments and strings"
- VS Code: "Rename Symbol" with full solution search

### 18.3 Common Pitfalls

| Pitfall | Mitigation |
|---------|------------|
| Missing string references | Search for quoted class names ("HealthBar") |
| Hardcoded log messages | Search for class names in log strings |
| XML documentation | Update `<see cref="..."/>` references |
| Test method names | Update test names to match new class names |

### 18.4 Verification Steps

1. **Build verification** - Full solution build after all renames
2. **Test execution** - Run all unit tests
3. **Reference check** - Search for old names, should return no results
4. **Documentation review** - Verify .md files updated
