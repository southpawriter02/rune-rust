# v0.3.22: The Tactician II (Combat Polish)

This milestone refines the combat experience by giving players control over the flow of information. It addresses the "wall of text" problem in the combat log and provides deep tactical insights into enemies, rewarding the "Knowledge as Power" pillar of the Jötun-Reader and Adept archetypes.

To manage the UI complexity and logic separation, this version is split into three sub-versions:
	•	v0.3.22a: The Filter (Log Customization) – Systems to filter, color-code, and manage combat log noise.
	•	v0.3.22b: The Inspector (Analysis Engine) – Logic to query enemy stats based on Bestiary knowledge levels.
	•	v0.3.22c: The Forecast (Predictive UI) – UI overlays showing hit chances and damage estimates before commitment.

## v0.3.22a: The Filter (Log Customization)
Goal: Implement granular control over the Combat Log, allowing players to toggle verbosity (e.g., hiding dice rolls, showing only dialogue) and creating a structured CombatLogEntry model to replace raw strings.
1. Architecture & Data Flow
	•	Model Upgrade: Replace List<string> CombatLog with List<CombatLogEntry>.
	•	Filtering Logic: The CombatScreenRenderer renders only entries that match the user's current filter settings (stored in GameSettings).
	•	Settings Integration: Add boolean flags to GameSettings (e.g., ShowDiceRolls, ShowFlavorText).
Data Flow: The Logging Pipeline
	1	Event: CombatService generates an event (e.g., AttackResult).
	2	Formatting: CombatLogFormatter creates a CombatLogEntry record containing:
	◦	Markup: The display text.
	◦	Category: Enum (Damage, Flavor, System, Dialogue).
	◦	Tags: List (e.g., "Critical", "Fire").
	3	Storage: Added to CombatViewModel.Log.
	4	Render:
	◦	Renderer checks GameSettings.
	◦	If entry.Category == Damage and !ShowDamageNumbers: Skip.
	◦	Else: Render.
2. Logic Decision Trees
A. Filter Check Logic Input: CombatLogEntry, Settings
	1	Category Switch:
	◦	Mechanics (Rolls/Damage): Check GameSettings.CombatVerbose.
	◦	Flavor (Barks/Descriptions): Check GameSettings.ShowFlavor.
	◦	System (Turn/Round): Always Show.
	2	Result: Render or Collapse.
3. Code Implementation

## v0.3.22b: The Inspector (Analysis Engine)
Goal: Implement the inspect command logic. This service queries the Bestiary (Data Capture System v0.1.3) to determine how much information the player "knows" about an enemy, returning a sanitized view of their stats.
1. Architecture & Data Flow
	•	Knowledge Query: AnalysisService checks the DataCaptureRepository for the enemy's Template ID.
	•	Masking: Based on the completion percentage (Stub, Partial, Complete, Mastered), the service hides or reveals fields.
	•	Archetype Bonus: Jötun-Readers (Adepts) treat all entries as one tier higher.
Data Flow: The Inspection Loop
	1	Command: inspect draugr.
	2	Resolution: AnalysisService.InspectEnemy(enemy).
	3	Tier Calculation:
	◦	Fetch CodexEntry for "Rusted Draugr".
	◦	Calculate % fragments found.
	4	Data Masking:
	◦	Stub (0%): Show Name, Description. HP: "Unknown".
	◦	Partial (50%): Show Max HP (~10% fuzz), Damage Type.
	◦	Complete (100%): Show Exact HP, Armor, Resistances.
	◦	Mastered: Show Abilities and AI Behavior hints.
	5	Output: Return EnemyIntel object to UI.
2. Logic Decision Trees
A. Knowledge Tier Logic Input: CompletionPct, Archetype
	1	Base Tier:
	◦	0-24%: Unknown.
	◦	25-74%: Obscured.
	◦	75-99%: Known.
	◦	100%: Mastered.
	2	Modifier: If Archetype == Adept/Jötun-Reader, Shift Tier +1 (Cap at Mastered).
	3	Result: Effective Tier.
3. Code Implementation
A. Analysis Service File: RuneAndRust.Engine/Services/AnalysisService.cs
public EnemyIntel Inspect(Combatant target)
{
    var entry = _codexRepo.GetByEntityId(target.TemplateId);
    var tier = CalculateKnowledgeTier(entry);

    return new EnemyIntel
    {
        Name = target.Name,
        // Fuzz HP if not fully known
        HpDisplay = tier >= KnowledgeTier.Known
            ? $"{target.CurrentHp}/{target.MaxHp}"
            : "???",
        Resistances = tier >= KnowledgeTier.Mastered
            ? target.Resistances
            : new List<string>()
    };
}

## v0.3.22c: The Forecast (Predictive UI)
Goal: Implement "Hover Intent" logic. When a player selects an action but hasn't confirmed it (e.g., highlighting "Attack"), the UI displays the probability and potential impact of that action.
1. Architecture & Data Flow
	•	Simulation: Use a lightweight version of AttackResolutionService (Dry Run) that calculates modifiers without rolling dice.
	•	Presentation: Update CombatViewModel to include a Prediction property.
Data Flow: The Prediction Loop
	1	Input: Player highlights "Attack" on "Goblin".
	2	Calculation:
	◦	Calculate Player Attack Dice vs Goblin Defense.
	◦	Estimate Hit Chance % (Statistical average).
	◦	Calculate Damage Range (Min/Max - Soak).
	3	Update: Set CombatViewModel.Prediction text: "Hit: 85% | Dmg: 4-10".
	4	Render: CombatScreenRenderer draws this info in the Footer/Tooltip area.

# Changelog: v0.3.22 - The Tactician II

**Release Date:** 2026-03-XX

## Summary
"The Tactician II" refines the combat interface, giving players the tools to analyze their foes and filter the noise of battle. Knowledge gathered via the Codex now directly impacts tactical clarity.

## Changes
- **Log Filtering:** Added options to hide dice rolls or flavor text in the combat log.
- **Inspect Command:** Use `inspect <target>` to view enemy stats. Detailed info requires unlocking Bestiary entries.
- **Combat Forecast:** Highlighting an action now displays estimated Hit Chance and Damage Range.
- **Jötun-Reader Bonus:** Adept characters automatically perceive higher-tier intel when inspecting enemies.
