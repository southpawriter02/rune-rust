Here is the comprehensive implementation plan for v0.2.1b: The Affliction (Status Effects).
This version introduces the temporal layer of combat—buffs, debuffs, and damage-over-time (DoT)—transforming the combat loop from a simple exchange of blows into a tactical resource management system.

v0.2.1b: The Affliction
Goal: Implement the StatusEffectService to manage the lifecycle (Application, Ticking, Expiration) of status effects, enforcing specific stacking rules (Intensify vs. Refresh) and mechanical consequences (Stat modifiers vs. Control).

1. Architecture & Data Flow
We introduce a dedicated service to manage the complexity of effect lifecycles, decoupling specific effect logic from the main CombatService.
Workflow Diagram: The Turn Lifecycle
	1	Turn Start: CombatService calls StatusEffectService.ProcessTurnStart(ActiveCombatant).
	◦	DoT Tick: Bleeding/Poisoned deal damage.
	◦	Duration Check: If Duration <= 0, remove effect.
	2	Action Check: CombatService checks StatusEffectService.CanAct(ActiveCombatant).
	◦	If Stunned: Log "Stunned!", skip Input/AI phase, auto-end turn.
	◦	If Clear: Proceed to Input (Player) or AI (Enemy).
	3	Action Execution: Attacks may call StatusEffectService.ApplyStatus().
	4	Stat Calculation: AttackResolutionService calls StatusEffectService.GetStatModifiers() to adjust Defense/Soak.
	5	Turn End: CombatService calls StatusEffectService.ProcessTurnEnd(ActiveCombatant).
	◦	Decrement: Reduce durations by 1.

2. Logic Decision Trees
A. Application Logic (ApplyStatus)
Input: Target, EffectType, Duration, Stacks
	1	Check Immunity: Is Target immune? (e.g., Skeleton vs Bleed)
	◦	Yes: Log "Immune" -> Exit.
	2	Check Existing: Does Target already have this EffectType?
	◦	No: Create new ActiveStatusEffect -> Add to Target -> Log "Applied" -> Exit.
	◦	Yes: Proceed to Stacking Rule.
	3	Apply Stacking Rule:
	◦	No Stack (e.g., Stunned): Reset Duration to max. (Do not increase intensity).
	◦	Refresh (e.g., Slowed): Reset Duration to max.
	◦	Intensify (e.g., Bleeding): Add Stacks (Cap at 5). Reset Duration to max.
	4	Log: "Effect Refreshed/Intensified".
B. Turn Start Processing
Input: Combatant
	1	Iterate Effects: For each active effect:
	◦	Bleeding: Deal 1d6 * Stacks Physical damage. Ignore Soak.
	◦	Poisoned: Deal 1d6 * Stacks Poison damage. Apply Soak.
	◦	Regeneration: Restore X HP.
	2	Log Outcomes: Return list of messages (e.g., "{Target} bleeds for 5 damage").

public class StatusEffectService : IStatusEffectService
{
    private readonly IDiceService _dice;
    private readonly ILogger<StatusEffectService> _logger;

    public void ApplyStatus(Combatant target, StatusEffectType type, int duration, Guid sourceId)
    {
        // TODO: Check Immunities (Future Phase)

        var existing = target.StatusEffects.FirstOrDefault(e => e.Type == type);
        if (existing != null)
        {
            HandleStacking(existing, duration);
        }
        else
        {
            target.StatusEffects.Add(new ActiveStatusEffect(type, duration, sourceId));
            _logger.LogInformation("Status Applied: {Type} on {Target}", type, target.Name);
        }
    }

    private void HandleStacking(ActiveStatusEffect effect, int newDuration)
    {
        // Spec-defined Stacking Rules
        switch (effect.Type)
        {
            case StatusEffectType.Bleeding:
            case StatusEffectType.Poisoned:
                effect.Stacks = Math.Min(5, effect.Stacks + 1); // Intensify
                break;
            default:
                // No Stack / Refresh behavior
                break;
        }
        effect.DurationRemaining = Math.Max(effect.DurationRemaining, newDuration);
    }

    public List<string> ProcessTurnStart(Combatant combatant)
    {
        var logs = new List<string>();

        foreach (var effect in combatant.StatusEffects)
        {
            if (effect.Type == StatusEffectType.Bleeding)
            {
                int dmg = _dice.RollSingle(6, "Bleed Tick") * effect.Stacks;
                combatant.CurrentHp -= dmg;
                logs.Add($"{combatant.Name} bleeds for {dmg} damage.");
            }
            // Add Poison logic here
        }
        return logs;
    }

    public bool CanAct(Combatant combatant)
    {
        return !combatant.StatusEffects.Any(e => e.Type == StatusEffectType.Stunned);
    }

    public void ProcessTurnEnd(Combatant combatant)
    {
        // Decrement durations
        for (int i = combatant.StatusEffects.Count - 1; i >= 0; i--)
        {
            var effect = combatant.StatusEffects[i];
            effect.DurationRemaining--;
            if (effect.DurationRemaining <= 0)
            {
                combatant.StatusEffects.RemoveAt(i);
                _logger.LogInformation("{Status} expired on {Target}", effect.Type, combatant.Name);
            }
        }
    }
}

# Changelog: v0.2.1b - The Affliction

**Release Date:** 2025-12-XX

## Summary
Introduced the Status Effect system, allowing for temporal combat mechanics including Damage over Time (DoT), Buffs, and Crowd Control.

## Features
- **Status Effect Engine:** Centralized service for managing effect lifecycles.
- **Bleeding:** Physical DoT that intensifies (stacks) and ignores armor.
- **Stunned:** Hard control effect that forces combatants to skip turns.
- **Fortified:** Defensive buff increasing Soak.
- **Stacking Rules:** Implemented distinct behaviors for "Intensify" (Bleed) vs "Refresh" (Stun).
- **Turn Integration:** Effects now tick damage at turn start and expire at turn end.

## Technical
- Added `IStatusEffectService` to DI container.
- Updated `Combatant` model to persist active effects.
- Extended `AttackResolutionService` to query active effects for stat modifiers.
