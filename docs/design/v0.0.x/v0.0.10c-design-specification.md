# v0.0.10c Design Specification: XML Documentation

**Version:** 0.0.10c
**Phase Name:** XML Documentation
**Parent Version:** v0.0.10 (Documentation & Polish)
**Prerequisites:** v0.0.9 Complete (Monster Variety & Loot)
**Estimated Tests:** ~0 (documentation-only phase)

---

## 1. Overview

### 1.1 Purpose

Ensure comprehensive XML documentation across all public APIs in the codebase, enabling IntelliSense support in IDEs, automated documentation generation, and improved developer experience for future contributors.

### 1.2 Current State

The codebase has partial XML documentation coverage:

| Layer | Current Coverage | Quality |
|-------|------------------|---------|
| Domain Entities | ~80% | Good - Player, Monster, Room, Item documented |
| Domain Value Objects | ~70% | Good - Stats, Position, most documented |
| Domain Definitions | ~90% | Excellent - ClassDefinition, AbilityDefinition well-documented |
| Domain Enums | ~60% | Partial - some members undocumented |
| Application Interfaces | ~90% | Good - IGameRepository, IGameRenderer documented |
| Application Services | ~50% | Partial - method docs incomplete |
| Application DTOs | ~70% | Good - record DTOs have param docs |
| Infrastructure | ~30% | Minimal - implementations lack detail |
| Presentation | ~20% | Minimal - views and adapters sparse |

### 1.3 Scope

**In Scope:**
- Complete XML documentation for all public types
- Add `<summary>` tags to all public members
- Add `<param>` tags to all method parameters
- Add `<returns>` tags to all non-void methods
- Add `<exception>` tags for documented exceptions
- Add `<remarks>` for complex behaviors
- Add `<example>` tags for key methods
- Add `<see cref=""/>` and `<seealso cref=""/>` cross-references
- Enable `GenerateDocumentationFile` in project files
- Enable XML documentation warnings

**Out of Scope:**
- Private/internal member documentation (optional, not enforced)
- Generated code documentation
- Test project documentation
- External documentation website generation (DocFX, etc.)

### 1.4 Design Goals

1. **IntelliSense Support**: All public APIs show helpful documentation in IDE
2. **Self-Documenting Code**: Code is understandable without external docs
3. **Consistency**: All documentation follows the same style and structure
4. **Accuracy**: Documentation accurately describes behavior
5. **Maintainability**: Documentation stays current with code changes
6. **Discoverability**: Cross-references help navigate related types

## 2. Documentation Standards

### 2.1 Required Tags by Member Type

| Member Type | Required Tags | Optional Tags |
|-------------|---------------|---------------|
| Class/Struct | `<summary>` | `<remarks>`, `<example>`, `<seealso>` |
| Interface | `<summary>`, `<remarks>` | `<example>`, `<seealso>` |
| Property | `<summary>` | `<value>`, `<remarks>`, `<example>` |
| Method | `<summary>`, `<param>`, `<returns>` | `<remarks>`, `<exception>`, `<example>` |
| Constructor | `<summary>`, `<param>` | `<exception>`, `<remarks>` |
| Enum | `<summary>` | `<remarks>` |
| Enum Member | `<summary>` | `<remarks>` |
| Record | `<summary>`, `<param>` (for primary constructor) | `<remarks>`, `<example>` |
| Event | `<summary>` | `<remarks>` |
| Delegate | `<summary>`, `<param>`, `<returns>` | `<remarks>` |

### 2.2 Documentation Style Guidelines

#### Summary Tag
- First sentence should be a complete, concise description
- Use third person (e.g., "Gets", "Sets", "Creates", not "Get", "Set", "Create")
- End with a period
- Avoid redundant phrases like "This method..." or "This property..."

```csharp
// GOOD
/// <summary>
/// Gets the player's current health points.
/// </summary>

// BAD
/// <summary>
/// This property gets the health of the player
/// </summary>
```

#### Parameter Tag
- Describe what the parameter represents, not its type
- Use "The" to start, then describe the parameter
- Note valid ranges or constraints if applicable

```csharp
// GOOD
/// <param name="damage">The raw damage amount before defense reduction.</param>
/// <param name="resourceTypeId">The resource type ID (e.g., "mana", "rage").</param>

// BAD
/// <param name="damage">An integer for damage.</param>
/// <param name="resourceTypeId">String parameter.</param>
```

#### Returns Tag
- Describe what is returned, including edge cases
- Mention null possibilities for reference types
- Use "A" or "The" to start

```csharp
// GOOD
/// <returns>The actual damage dealt after defense calculation.</returns>
/// <returns>The ability definition, or <c>null</c> if not found.</returns>
/// <returns><c>true</c> if the item was added; otherwise, <c>false</c>.</returns>

// BAD
/// <returns>Returns damage.</returns>
/// <returns>AbilityDefinition</returns>
```

#### Exception Tag
- Document exceptions that callers should handle
- Include the condition that causes the exception
- Use `cref` to reference the exception type

```csharp
/// <exception cref="ArgumentNullException">
/// Thrown when <paramref name="player"/> is null.
/// </exception>
/// <exception cref="ArgumentOutOfRangeException">
/// Thrown when <paramref name="damage"/> is negative.
/// </exception>
```

#### Remarks Tag
- Use for additional context beyond the summary
- Explain implementation details, algorithms, or behavior
- Use lists for multi-step processes

```csharp
/// <remarks>
/// <para>Combat proceeds as follows:</para>
/// <list type="number">
/// <item>Player attacks the monster, dealing damage based on attack vs defense.</item>
/// <item>If the monster survives, it counterattacks the player.</item>
/// <item>The result indicates damage dealt/received and whether either was defeated.</item>
/// </list>
/// </remarks>
```

#### Example Tag
- Provide concise, runnable code examples
- Focus on common use cases
- Keep examples simple and focused

```csharp
/// <example>
/// <code>
/// var player = new Player("Hero");
/// var damage = player.TakeDamage(15);
/// Console.WriteLine($"Took {damage} damage, now at {player.Health} HP");
/// </code>
/// </example>
```

### 2.3 Cross-Reference Guidelines

Use `<see cref=""/>` for inline references:
```csharp
/// <summary>
/// Gets the player's core attributes. See <see cref="PlayerAttributes"/> for details.
/// </summary>
```

Use `<seealso cref=""/>` for related types:
```csharp
/// <seealso cref="Monster"/>
/// <seealso cref="CombatService"/>
```

Use `<paramref name=""/>` to reference parameters:
```csharp
/// <param name="damage">The raw damage amount.</param>
/// <returns>The actual damage after <paramref name="damage"/> is reduced by defense.</returns>
```

## 5. Project Configuration

### 5.1 Enable Documentation Generation

Add to each `.csproj` file:

```xml
<PropertyGroup>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
  <NoWarn>$(NoWarn);1591</NoWarn> <!-- Remove to enforce documentation warnings -->
</PropertyGroup>
```

### 5.2 Enable Documentation Warnings (Optional)

To enforce documentation, remove `1591` from `NoWarn`:

```xml
<PropertyGroup>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
  <!-- CS1591: Missing XML comment for publicly visible type or member -->
  <!-- Uncomment below to enforce documentation -->
  <!-- <TreatWarningsAsErrors>true</TreatWarningsAsErrors> -->
</PropertyGroup>
```

### 5.3 Warning Codes Reference

| Code | Description | Enforcement |
|------|-------------|-------------|
| CS1591 | Missing XML comment for publicly visible type or member | Optional |
| CS1572 | XML comment has a param tag, but there is no parameter by that name | Always fix |
| CS1573 | Parameter has no matching param tag in the XML comment | Optional |
| CS1574 | XML comment has cref attribute that could not be resolved | Always fix |
| CS1584 | XML comment has syntactically incorrect cref attribute | Always fix |
| CS1587 | XML comment is not placed on a valid language element | Always fix |
| CS1589 | Unable to include XML fragment | Always fix |
| CS1590 | Invalid XML include element | Always fix |

## 6. Documentation Examples

### 6.1 Well-Documented Entity Example

```csharp
using RuneAndRust.Domain.Interfaces;
using RuneAndRust.Domain.ValueObjects;

namespace RuneAndRust.Domain.Entities;

/// <summary>
/// Represents a player character in the game.
/// </summary>
/// <remarks>
/// <para>
/// The Player entity encapsulates all character state including health, stats,
/// position in the dungeon, and inventory. It handles combat damage, healing,
/// and item management.
/// </para>
/// <para>
/// Players are created via the constructor and managed by <see cref="GameSession"/>.
/// State changes should be made through the provided methods to ensure validity.
/// </para>
/// </remarks>
/// <seealso cref="Monster"/>
/// <seealso cref="GameSession"/>
public class Player : IEntity
{
    /// <summary>
    /// Gets the unique identifier for this player.
    /// </summary>
    public Guid Id { get; private set; }

    /// <summary>
    /// Gets the player's display name.
    /// </summary>
    /// <value>A non-null, non-empty string.</value>
    public string Name { get; private set; }

    /// <summary>
    /// Gets the player's current health points.
    /// </summary>
    /// <value>
    /// A non-negative integer. Zero means the player is dead.
    /// Maximum value is <see cref="Stats"/>.<see cref="Stats.MaxHealth"/>.
    /// </value>
    public int Health { get; private set; }

    /// <summary>
    /// Gets the player's base statistics.
    /// </summary>
    /// <seealso cref="Stats"/>
    public Stats Stats { get; private set; }

    /// <summary>
    /// Gets a value indicating whether the player is alive.
    /// </summary>
    /// <value><c>true</c> if <see cref="Health"/> is greater than zero; otherwise, <c>false</c>.</value>
    public bool IsAlive => Health > 0;

    /// <summary>
    /// Private parameterless constructor for Entity Framework Core.
    /// </summary>
    private Player()
    {
        Name = null!;
    }

    /// <summary>
    /// Creates a new player with the specified name and optional stats.
    /// </summary>
    /// <param name="name">The player's display name. Must not be null or empty.</param>
    /// <param name="stats">
    /// Optional custom stats. If <c>null</c>, <see cref="Stats.Default"/> is used.
    /// </param>
    /// <exception cref="ArgumentNullException">
    /// Thrown when <paramref name="name"/> is <c>null</c>.
    /// </exception>
    /// <example>
    /// <code>
    /// // Create player with default stats
    /// var player = new Player("Hero");
    ///
    /// // Create player with custom stats
    /// var warrior = new Player("Warrior", new Stats(150, 15, 10));
    /// </code>
    /// </example>
    public Player(string name, Stats? stats = null)
    {
        Id = Guid.NewGuid();
        Name = name ?? throw new ArgumentNullException(nameof(name));
        Stats = stats ?? Stats.Default;
        Health = Stats.MaxHealth;
    }

    /// <summary>
    /// Applies damage to the player, reduced by their defense stat.
    /// </summary>
    /// <param name="damage">The raw damage amount before defense reduction.</param>
    /// <returns>The actual damage dealt after defense calculation.</returns>
    /// <exception cref="ArgumentOutOfRangeException">
    /// Thrown when <paramref name="damage"/> is negative.
    /// </exception>
    /// <remarks>
    /// <para>
    /// Damage is calculated as: <c>max(0, damage - Defense)</c>.
    /// The player's health cannot go below zero.
    /// </para>
    /// </remarks>
    /// <example>
    /// <code>
    /// var player = new Player("Hero", new Stats(100, 10, 5));
    /// var actualDamage = player.TakeDamage(20); // Returns 15 (20 - 5 defense)
    /// </code>
    /// </example>
    public int TakeDamage(int damage)
    {
        if (damage < 0)
            throw new ArgumentOutOfRangeException(nameof(damage), "Damage cannot be negative");

        var actualDamage = Math.Max(0, damage - Stats.Defense);
        Health = Math.Max(0, Health - actualDamage);
        return actualDamage;
    }
}
```

### 6.2 Well-Documented Service Example

```csharp
using Microsoft.Extensions.Logging;
using RuneAndRust.Application.Interfaces;
using RuneAndRust.Domain.Definitions;
using RuneAndRust.Domain.Entities;

namespace RuneAndRust.Application.Services;

/// <summary>
/// Manages class definitions and player class assignment.
/// </summary>
/// <remarks>
/// <para>
/// The ClassService provides methods for retrieving class definitions and
/// assigning classes to players during character creation.
/// </para>
/// <para>
/// Classes are loaded from configuration via <see cref="IGameConfigurationProvider"/>.
/// </para>
/// </remarks>
/// <seealso cref="ClassDefinition"/>
/// <seealso cref="ArchetypeDefinition"/>
public class ClassService
{
    private readonly IGameConfigurationProvider _configProvider;
    private readonly ILogger<ClassService> _logger;

    /// <summary>
    /// Creates a new instance of <see cref="ClassService"/>.
    /// </summary>
    /// <param name="configProvider">The configuration provider for class definitions.</param>
    /// <param name="logger">The logger for diagnostic output.</param>
    /// <exception cref="ArgumentNullException">
    /// Thrown when <paramref name="configProvider"/> or <paramref name="logger"/> is <c>null</c>.
    /// </exception>
    public ClassService(
        IGameConfigurationProvider configProvider,
        ILogger<ClassService> logger)
    {
        _configProvider = configProvider ?? throw new ArgumentNullException(nameof(configProvider));
        _logger = logger ?? throw new ArgumentNullException(nameof(logger));
    }

    /// <summary>
    /// Gets a class definition by its unique identifier.
    /// </summary>
    /// <param name="classId">The class identifier (case-insensitive).</param>
    /// <returns>
    /// The <see cref="ClassDefinition"/> if found; otherwise, <c>null</c>.
    /// </returns>
    /// <example>
    /// <code>
    /// var shieldmaiden = classService.GetClass("shieldmaiden");
    /// if (shieldmaiden != null)
    /// {
    ///     Console.WriteLine($"Found class: {shieldmaiden.Name}");
    /// }
    /// </code>
    /// </example>
    public ClassDefinition? GetClass(string classId)
    {
        _logger.LogDebug("GetClass called for: {ClassId}", classId);
        return _configProvider.GetClassById(classId);
    }

    /// <summary>
    /// Assigns a class to a player, applying stat modifiers and starting abilities.
```

## 7. Acceptance Criteria

### 7.1 Documentation Coverage

- [ ] All public classes have `<summary>` tags
- [ ] All public interfaces have `<summary>` and `<remarks>` tags
- [ ] All public methods have `<summary>`, `<param>`, and `<returns>` tags
- [ ] All public properties have `<summary>` tags
- [ ] All public constructors have `<summary>` and `<param>` tags
- [ ] All enum members have `<summary>` tags
- [ ] All record parameters have documentation in the record's `<param>` tags

### 7.2 Documentation Quality

- [ ] Summaries are concise and accurate
- [ ] Parameters describe meaning, not type
- [ ] Return values describe all possibilities including null
- [ ] Exceptions are documented with conditions
- [ ] Complex methods have `<remarks>` with additional detail
- [ ] Key methods have `<example>` code snippets
- [ ] Cross-references use `<see cref=""/>` correctly

### 7.3 Project Configuration

- [ ] All projects have `GenerateDocumentationFile` enabled
- [ ] XML documentation files are generated during build
- [ ] Documentation warnings are addressed or explicitly suppressed

### 7.4 IDE Support

- [ ] IntelliSense shows documentation for all public APIs
- [ ] Parameter hints show documentation
- [ ] Quick info popups display full documentation
