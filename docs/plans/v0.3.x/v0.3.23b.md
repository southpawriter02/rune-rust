# v0.3.23b: The Loop (Event-Driven Architecture)

**Status:** Planned
**Parent Plan:** [v0.3.23](./v0.3.23.md)
**Context:** This sub-plan transforms the game loop from a blocking "wait for key" model to a non-blocking loop that processes input if available, but otherwise updates background systems (Ambience, VFX).

## 1. Loop Architecture

### 1.1. Modify TerminalGameLoop
Locate `RuneAndRust.Terminal/Services/TerminalGameLoop.cs`.

**Current Logic (Estimated):**
```csharp
while (running) {
    _inputHandler.HandleInput(); // Blocks here
    // ...
}
```

**New Logic:**
- Inject `IInputService`, `IAmbienceService` (if available), `IVisualEffectService` (if available).
- Use `CancellationToken`.

```csharp
public async Task RunAsync(CancellationToken ct)
{
    while (!ct.IsCancellationRequested)
    {
        // 1. Process Input (Non-blocking check)
        if (_inputService.IsInputAvailable())
        {
            var inputEvent = _inputService.ReadNext();
            await HandleInputAsync(inputEvent);
        }

        // 2. Background Updates
        // Only update if enough time has passed or explicitly designed for per-tick updates
        await _ambienceService.UpdateAsync();

        // 3. Throttle
        await Task.Delay(20, ct); // ~50 ticks per second
    }
}
```

## 2. Service Updates

### 2.1. Update InputService
Ensure `IsInputAvailable()` is implemented in `RuneAndRust.Engine/Services/InputService.cs`.
- Implementation: `return Console.KeyAvailable;`

### 2.2. Ambience Service Ticking
Ensure `IAmbienceService` has an `UpdateAsync()` method (or similar) that can safely be called every tick.
- If it plays audio/loops, it might not need per-tick updates, but if it handles dynamic fading or procedural sound triggering, it does.

## 3. Integration

### 3.1. Main Entry Point
Update `Program.cs` to pass a `CancellationToken` to the game loop.
- Handle `Ctrl+C` (SIGINT) to cancel the token gracefully.

## 4. Verification

### 4.1. Responsiveness Test
- Run the game.
- Ensure typing feels responsive (no input lag from `Task.Delay`).
- Ensure the game doesn't consume 100% CPU (verify via Task Manager/top if possible, or just trust the `Task.Delay`).

### 4.2. Background Processes
- If `AmbienceService` has visual components (e.g., status bar ticker, time updates), verify they update without player input.
