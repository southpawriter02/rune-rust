## Overview

**v0.4.4e** completes the Mystic experience by implementing the persistent knowledge system ("The Grimoire"). Unlike the raw mechanics of casting (v0.4.4a/b), this sub-version focuses on **acquisition and progression**.

It introduces a crucial progression mechanic: **Ranked Spells**. Every spell can be upgraded through three ranks using **Progression Points (PP)**, transforming simple effects into potent tools. The Grimoire UI serves as the interface for this study, allowing players to learn new spells, deepen their understanding of known ones, and prepare their mind (Quick Slots) for combat.

### Relationship to Previous Phases

| Phase | Dependency | Description |
|-------|------------|-------------|
| **v0.4.4a** | Resonance | Spell costs/effects are modified by Resonance state |
| **v0.4.4c** | Mastery | School Mastery Tier (Novice-Archon) acts as a *gate* for learning spells |
| **v0.4.1** | Progression | Uses the standard PP currency for purchasing/upgrading |

### The Progression Loop

1. **Earn XP (Saga):** Gain Levels → Earn Progression Points (PP).
2. **Practice (Mastery):** Cast spells → Gain School Mastery XP → Unlock Tiers (e.g., Apprentice).
3. **Study (Grimoire):** Spend PP to learn new spells (gate: Mastery Tier) or upgrade known spells (gate: Rank).

## Phase A: The Data Structure (Ranked Spells)

**Goal:** Establish the data models required to support multi-rank spells and persistent character knowledge.

### 1. Architecture & Data Flow

We introduce `SpellDefinition` as a rich, ranked data structure separate from the flat `ActiveAbility`.

**Components:**
- **SpellDefinition (Core):** Contains metadata and a dictionary of `SpellRankData` (Rank 1-3).
- **SpellRankData (Core):** Defines the specific costs, damage, and scripts for a rank.
- **CharacterSpell (Entities):** Junction table tracking which spells a character knows and their current rank.

**Workflow: Fetching Spell Data**
1. System requests spell info for `Fireball` at `Rank 2`.
2. `SpellRegistry` looks up `SpellDefinition`.
3. Returns `SpellRankData` for Rank 2 (e.g., "3d6 Damage", Cost: 5 AP).

**Runtime Conversion (Adapter Pattern)**
The Combat Engine expects `ActiveAbility` objects. `GrimoireService` or `MagicService` provides an adapter method:
`GetActiveAbility(Character, SpellId)` -> looks up Rank -> maps `SpellRankData` to a transient `ActiveAbility` with the correct script/costs.

### 2. Code Implementation

#### A. Spell Models (Core/Models/Magic)

```csharp
public class SpellDefinition
{
    public Guid Id { get; set; }
    public string Name { get; set; } = string.Empty;
    public string Description { get; set; } = string.Empty;
    public SpellSchool School { get; set; }

    // Key: Rank (1, 2, 3), Value: Data for that rank
    public Dictionary<int, SpellRankData> Ranks { get; set; } = new();

public class SpellRankData
{
    public int Rank { get; set; }
    public int ApCost { get; set; }
    public int FluxCost { get; set; }
    public string EffectScript { get; set; } = string.Empty; // e.g., "DAMAGE:Fire:3d6"
    public string UpgradeDescription { get; set; } = string.Empty; // e.g., "+1d6 Damage"
    public int UpgradeCostPP { get; set; } // Cost to reach this rank (Rank 1 cost = purchase cost)
}
```

## Phase B: The Logic (Grimoire Service)

**Goal:** Implement the rules for learning, upgrading, and validating spells.

### 1. Logic Decision Trees

#### A. Learn Spell (Purchase Rank 1)
**Input:** `Character`, `SpellId`

1. **Check Requirements:**
   - Character has `SpellId`? -> Error: "Already known".
   - Character PP >= `Rank1Cost`? -> Error: "Not enough PP".
   - School Mastery Tier >= `Spell.Tier`? -> Error: "Mastery too low".
2. **Execute:**
   - Deduct PP.
   - Create `CharacterSpell` (Rank 1).
   - Grant School Mastery XP (Bonus for learning).
   - Publish `SpellLearnedEvent`.

#### B. Upgrade Spell (Rank X -> X+1)
**Input:** `Character`, `SpellId`

1. **Check Requirements:**
   - Character knows spell? -> Error: "Unknown spell".
   - Current Rank < Max Rank (3)? -> Error: "Max rank reached".
   - Character PP >= `NextRankCost`? -> Error: "Not enough PP".
2. **Execute:**
   - Deduct PP.
   - Increment `CharacterSpell.Rank`.
   - Publish `SpellUpgradedEvent`.

### 2. Code Implementation

#### A. Grimoire Service (Engine/Services)

**File:** `RuneAndRust.Engine/Services/GrimoireService.cs`

```csharp
public class GrimoireService : IGrimoireService
{
    private readonly ILogger<GrimoireService> _logger;
    private readonly IMasteryService _masteryService;

    public Result LearnSpell(Character character, Guid spellId)
    {
        var spellDef = _spellRegistry.Get(spellId);

        // 1. Check Mastery Gate
        var mastery = _masteryService.GetMastery(character, spellDef.School);
        if (mastery.Tier < spellDef.Tier)
            return Result.Failure($"Requires {spellDef.School} Tier {spellDef.Tier}");

        // 2. Check Cost
        int cost = spellDef.Ranks[1].UpgradeCostPP;
        if (character.ProgressionPoints < cost)
            return Result.Failure("Insufficient Progression Points");

        // 3. Execute
        character.ProgressionPoints -= cost;
        character.KnownSpells.Add(new CharacterSpell { SpellId = spellId, Rank = 1 });

        _logger.LogInformation("Character {Id} learned {Spell} (Rank 1)", character.Id, spellDef.Name);
        return Result.Success();
    }
}
```

### Summary
Completing the Mystic archetype, **The Grimoire** introduces the persistent knowledge layer. Players can now invest Progression Points to learn and upgrade spells through three ranks of power, managed via a dedicated TUI.

### Features
- **Ranked Spells:** All spells now have 3 ranks. Upgrading improves damage, reduces cost, or adds effects.
- **Grimoire UI:** A new management screen (`grimoire`) to view known spells, check mastery requirements, and spend PP.
- **Quick Slots:** Bind up to 5 favorite spells to number keys `1-5` for rapid access in combat.
- **Mystic Progression:** Integration with School Mastery tiers to gate powerful spells.

### Technical
- **New Models:** `SpellDefinition` (Ranked), `CharacterSpell`.
- **New Service:** `GrimoireService` handling logic for PP spending and validation.
- **Persistence:** Database migration for `KnownSpells` and `QuickSlots`.
- **Domain 4 Compliance:** Flavor text for upgrades emphasizes "deepening understanding" rather than "leveling up".
