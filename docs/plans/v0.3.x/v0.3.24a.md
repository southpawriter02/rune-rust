# Implementation Plan - v0.3.24a: The Broom (Deprecation Cleanup)

**Status:** Planned
**Version:** v0.3.24a
**Milestone:** v0.3.24 (The Precursor)

## Goal
Remove all code paths marked `[Obsolete]`, delete temporary test assets (specifically `GenerateTestMapAsync`), and ensure debug commands are stripped or locked behind `#if DEBUG` directives. This step ensures the codebase is clean and secure before the v0.4.0 Alpha release.

## 1. Deprecation: Remove `GenerateTestMapAsync`

The `GenerateTestMapAsync` method in `DungeonGenerator` is a legacy v0.0.5 artifact used for testing. It must be removed and replaced with calls to `GenerateDungeonAsync("the_roots")`.

### 1.1 Update `Program.cs`
- **File:** `RuneAndRust.Terminal/Program.cs`
- **Change:**
  - Locate usage of `GenerateTestMapAsync()`.
  - Replace with `GenerateDungeonAsync("the_roots")`.

### 1.2 Update Tests
- **File:** `RuneAndRust.Tests/Engine/DungeonGeneratorTests.cs`
- **Change:**
  - Refactor tests relying on `GenerateTestMapAsync` to use `GenerateDungeonAsync("the_roots")`.
  - If a specific test shape is required, implement a `TestDungeonHelper` in the test project to manually construct the graph, rather than relying on production code.

### 1.3 Remove Method
- **File:** `RuneAndRust.Engine/Services/DungeonGenerator.cs`
- **Action:** Delete `GenerateTestMapAsync`.

## 2. Cleanup: "Training Dummy" Removal

The "Training Dummy" is a hardcoded entity used for debug combat. It should not exist in production logic.

### 2.1 Update `CommandParser`
- **File:** `RuneAndRust.Engine/Services/CommandParser.cs`
- **Change:**
  - Wrap the `debug-combat` command logic in `#if DEBUG` preprocessor directives.
  - Ensure users in Release builds receive an "Unknown command" response.

### 2.2 Verify `Enemy.cs`
- **File:** `RuneAndRust.Core/Entities/Enemy.cs`
- **Change:**
  - Review the default value of `Name` property. If it defaults to "Training Dummy", change it to "Unknown Enemy" or make it required/constructor-injected to prevent accidental creation of dummy entities.

### 2.3 Update Tests
- **File:** `RuneAndRust.Tests/Infrastructure/TestGameHost.cs`
- **File:** `RuneAndRust.Tests/Engine/CombatServiceTests.cs`
- **Change:**
  - Ensure tests explicitly create a "Training Dummy" enemy using the `EnemyFactory` or manual instantiation, rather than relying on implicit defaults.

## 3. Security: Lock Down Debug Services

Debug tools must not be accessible in the Release build.

### 3.1 Conditional Service Registration
- **File:** `RuneAndRust.Terminal/Program.cs`
- **Change:**
  - Wrap registration of `IDebugConsoleService` and `ICheatService` in `#if DEBUG`.
  - Wrap registration of `DebugConsoleRenderer` in `#if DEBUG`.

### 3.2 Update `CommandParser` Injection
- **File:** `RuneAndRust.Engine/Services/CommandParser.cs`
- **Change:**
  - Update constructor to handle potentially null `ICheatService` or wrap the dependency usage in `#if DEBUG`. Alternatively, inject `IEnumerable<ICheatService>` and it will be empty in Release.
  - **Preferred:** Wrap the private field and constructor parameter in `#if DEBUG`.

## 4. Housekeeping: Console.WriteLine Audit

Direct `Console.WriteLine` usage bypasses the logging system and TUI rendering.

### 4.1 `EmergencySaveService`
- **File:** `RuneAndRust.Engine/Services/EmergencySaveService.cs`
- **Change:**
  - Evaluate if `Console.WriteLine` is necessary (it might be if the TUI has crashed).
  - If TUI is crashed, `Console.WriteLine` is acceptable as a fallback. Add a comment `// INTENTIONAL: Fallback output during crash`.

### 4.2 Other Occurrences
- **Action:** grep for `Console.WriteLine` in `RuneAndRust.Engine` and `RuneAndRust.Core`.
- **Change:** Replace with `ILogger` or `IOutputHandler` where appropriate.

## 5. Verification Steps

1.  **Build Release:** Run `dotnet build -c Release`.
2.  **Verify Cheats:** Start the game and type `/god` or `debug-combat`. It should fail silently or show "Unknown command".
3.  **Run Tests:** Run `dotnet test`. Ensure all tests pass after removing `GenerateTestMapAsync`.
